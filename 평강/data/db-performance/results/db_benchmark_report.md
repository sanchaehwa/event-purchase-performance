# DB 성능 비교 보고서

## 1. 개요

이 문서는 로컬 Docker 환경에서 MySQL, MariaDB, PostgreSQL, MongoDB를 대상으로 상품 데이터 조회 성능을 비교한 결과를 정리한 문서다.

- 1만 건 결과 파일: `C:\Users\leepg\OneDrive\바탕 화면\2026\멋쟁이사자_백엔드\event-purchase-performance\평강\results\performance_results.csv`
- 10만 건 결과 파일: `C:\Users\leepg\OneDrive\바탕 화면\2026\멋쟁이사자_백엔드\event-purchase-performance\평강\db-performance\results\result2.csv`
- 사용 스크립트:
  - `scripts/generate_scaled_csv.py`
  - `scripts/bulk_insert.py`
  - `scripts/run_tests.py`

## 2. 테스트 방식과 조건

### 환경

- 실행 환경: 로컬 Windows PC + Docker 컨테이너
- DB 구성
  - MySQL: `localhost:3308`
  - MariaDB: `localhost:3309`
  - PostgreSQL: `localhost:5433`
  - MongoDB: `localhost:27017`

### 데이터셋

- 원본 CSV: `result/ably_products.csv`
- 1만 건 테스트: 원본 CSV 그대로 사용
- 10만 건 테스트: 원본 1만 건 CSV를 10배 반복해 `ably_products_100k.csv` 생성 후 적재
- 적재 대상
  - RDB: `products` 테이블
  - MongoDB: `products` 컬렉션

### 측정 방식

- 워밍업: 10회
- 본 측정: 100회
- 측정 지표: `avg`, `min`, `max`, `stddev`
- 본 문서 표는 `avg(ms)` 기준으로 정리
- 캐시/플랜 초기화 방식
  - MySQL, MariaDB: `FLUSH TABLES`
  - PostgreSQL: `DISCARD ALL`
  - MongoDB: `planCacheClear`

### 테스트 항목

- 단순조회: ID 단건 조회
- 전체조회: 전체 상품 조회
- 필터조회: 카테고리 + 가격 범위
- 정렬조회: 가격 오름차순, 리뷰 많은 순
- 집계: 브랜드별 평균 가격, 카테고리별 상품 수
- 복합조건: 카테고리 + 브랜드 + 가격 범위 + 정렬 + LIMIT
- 인덱스 전/후 필터 및 복합조건
- JOIN vs Lookup
- 트랜잭션 성격의 연속 UPDATE 2건

### 해석 시 주의점

- 본 테스트는 로컬 단일 머신 기준이다. 실제 서비스 네트워크, 앱 서버, ORM, 직렬화 비용은 포함되지 않았다.
- 10만 건 데이터는 실데이터 10만 건이 아니라 1만 건을 반복 확장한 데이터다.
- MongoDB의 `트랜잭션_UPDATE2건`은 세션 기반 트랜잭션이 아니라 연속 update 2회 성능이다.
- RDB의 `인덱스없음_필터조회`는 기존 인덱스가 남아 있었을 가능성이 있어 참고용으로 해석하는 것이 안전하다.

- MariaDB는 PostgreSQL만큼 강하지는 않았지만 전반적으로 안정적이었다.
- MySQL은 대부분 항목에서 성능 하락 폭과 절대 시간 모두 불리했다.

## 6. 대규모 트래픽 이커머스 시스템에서 어떤 DB가 유리한가

### 1순위 추천: PostgreSQL

대규모 트래픽 쇼핑몰의 핵심 쿼리는 보통 다음과 같다.

- 카테고리/가격/브랜드 필터
- 정렬 조건이 붙은 상품 목록
- 리뷰 수, 평점, 가격 기준 정렬
- 집계와 통계성 조회
- 주문, 재고, 결제와 연결되는 정합성 중심 트랜잭션

이번 테스트 기준으로는 PostgreSQL이 이런 유형에서 가장 강했다.

- 10만 건 필터조회 63.038ms
- 10만 건 정렬조회_가격낮은순 25.100ms
- 10만 건 정렬조회_리뷰많은순 19.780ms
- 10만 건 집계_브랜드별평균가격 54.693ms
- 10만 건 JOIN_브랜드통계 60.364ms
- 10만 건 복합조건 7.656ms

즉, 상품 목록 조회와 운영성 집계가 많은 전형적인 이커머스 백엔드에서는 PostgreSQL을 메인 DB로 두는 편이 가장 설득력 있다.

### 2순위 대안: MariaDB

MariaDB는 10만 건에서 PostgreSQL 다음으로 무난했다.

- 정렬, 필터, 트랜잭션이 비교적 안정적임
- MySQL 계열이라 익숙한 팀에 진입 장벽이 낮음
- 다만 집계와 JOIN, 대용량 필터/정렬은 PostgreSQL보다 다소 밀림

운영 팀이 MySQL 계열에 익숙하고, PostgreSQL 도입 비용이 부담될 때 대안으로 적합하다.

### 보조 저장소로 적합: MongoDB

MongoDB는 다음 조건에서는 장점이 있다.

- 단건 조회가 빠름
- 전체 document 읽기 속도가 빠름
- 스키마 유연성이 좋음

하지만 이번 테스트에서는 데이터가 커질수록 다음 항목에서 약점이 분명했다.

- 필터조회 245.414ms
- 정렬조회 175ms~188ms
- 집계 147ms~434ms
- Lookup 512.135ms
- 복합조건 287.712ms

따라서 MongoDB를 쇼핑몰 메인 트랜잭션 DB로 단독 운영하는 것은 이번 실험 기준으로는 비추천이다. 대신 아래와 같은 보조 역할은 충분히 고려할 수 있다.

- 유연한 상품 속성 저장
- 로그, 이벤트, 추천 피드 저장
- 캐시성 document 조회

### 이번 실험 기준 비추천: MySQL

MySQL도 사용 자체가 불가능한 것은 아니지만, 이번 로컬 Docker 실험에서는 전반적으로 가장 불리했다.

- 10만 건 전체조회 3705.221ms
- 10만 건 필터조회 177.897ms
- 10만 건 집계_브랜드별평균가격 313.893ms
- 10만 건 JOIN_브랜드통계 326.561ms

즉, 같은 조건에서는 MariaDB나 PostgreSQL보다 우선순위가 낮다.

## 7. 최종 결론

이번 실험만 기준으로 정리하면 다음과 같다.

1. 1만 건에서는 MongoDB가 단순 조회에서 강했고 PostgreSQL이 정렬/집계에서 강했다.
2. 10만 건으로 커지자 PostgreSQL이 쇼핑몰 핵심 쿼리 대부분에서 가장 유리해졌다.
3. MariaDB는 전체적으로 안정적인 2순위 선택지였다.
4. MongoDB는 단건/문서 조회에는 강하지만, 대규모 상품 필터링/정렬/집계 중심의 메인 DB로는 불리했다.
5. 대규모 트래픽 이커머스 시스템의 메인 DB는 PostgreSQL이 가장 적합하다는 결론이 나온다.

실무적으로는 `PostgreSQL을 메인 DB로 사용하고`, 필요하면 `MongoDB를 보조 저장소로 병행`하는 구조가 가장 현실적이다.
